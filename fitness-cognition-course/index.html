The Big Question

Does how much you exercise today predict how well your brain works tomorrow?

A research project hiding inside Python code

This codebase is an extension of a published scientific paper — Manning et al. (2022) — that asked a simple but profound question:

🧠
The core question: Can we use wearable fitness data — steps, calories, sleep, BMI — to predict how well someone will perform on memory and cognitive tests?

The original paper found correlations using simple statistics. This project goes further: it trains AI models to see whether fitness data from weeks before a brain test can predict performance — suggesting exercise doesn't just correlate with cognition, it may actually cause it.

📊

The Input

Daily fitness readings from wearables: steps, calories burned, active minutes, weight, BMI — tracked for hundreds of days per person.

🧩

The Output

Scores on cognitive tasks: free recall accuracy, temporal clustering, semantic clustering — tests that measure memory and how the brain organizes information.

⏱️

The Twist

Does fitness from 30 days ago predict today's brain test score? 14 days? 90? This lag analysis is the project's key scientific contribution.

02

Cleaning the Mess

Raw wearable data is like a diary written in crayon in the rain — beautiful intentions, brutal readability.

The problem with real-world data

Think of wearable data like a paper attendance sheet for a class of 100 people. Some days people forgot to sign in. Some people wrote "0" when they meant "didn't take my watch off." Some people missed entire weeks. The raw data is full of holes and errors — and an AI fed garbage will learn garbage.

📝
What the raw data looks like: Each participant has a CSV file with one row per day per measurement. But there are gaps (missed days), impossible values (weight = 0), and no guarantee every feature was recorded on the same days.

The file reading_data.py runs a strict 7-step pipeline to fix all of this before a single number touches the AI model.

💡
Vocabulary moment: This process — loading, filtering, fixing, and transforming raw data — is called data preprocessing. It's often 80% of the work in any machine learning project.

The 13 features — what gets measured

The features are the 13 health metrics tracked for each participant, each day:

CODE 
FEATURE_COLS = [
    "bmi", "bodyfat", "cal", "cal_bmr",
    "distance", "fair_act_mins",
    "food_cal_log", "light_act_mins",
    "sed_mins", "steps",
    "very_act_mins", "water_log", "weight",
]
PLAIN ENGLISH

This is a list of the 13 columns we care about.

Body measurements: body mass index, body fat percentage, calorie burn, basal metabolic rate.

Activity: distance traveled, fairly-active minutes.

Nutrition & light movement: food calories logged, light-active minutes.

Sedentary minutes, steps taken.

Very-active minutes, water logged, and current weight.

End of the list.

These are grouped into three categories, which matters for how gaps get filled later:

🏃

Activity Metrics

Steps, distance, calories, active minutes. If you didn't log activity, it was probably 0 — so missing values get filled with zero.

⚖️

Physical State

Weight, BMI, body fat. These don't change overnight — so missing days get filled by carrying the last known value forward.

🥗

Nutrition

Food calories, water. Like activity — if you didn't log it, zero is the safe assumption.

The 7-step pipeline in action

Imagine the raw data walking into a hospital for a check-up. Here's what each step does to it:

📁
Raw Data

Hi, I just arrived! I have 300 days for BFM_AMT_0005 but... I'm missing Feb 4-12 entirely. Also some of my weight values are 0.0, and my dates aren't in a consistent format.

🔍
Step 0: Quality Gate

First, I check: does this participant have at least 100 days where 5+ features were recorded? If not, they get dropped from the study entirely. Sorry, but garbage-in, garbage-out.

📅
Step 1: Daily Resampling

I insert explicit NaN rows for every missing date. So instead of jumping from Feb 3 to Feb 13, you now have 10 empty rows. Gaps become visible — not hidden.

🔧
Step 2: Fix Impossible Zeros

Nobody weighs 0 kg. I convert all zeros in weight, BMI, and body fat to NaN (empty). They're clearly recording errors, not real measurements.

⏩
Step 3: Forward Fill Physical Metrics

For weight, BMI, and body fat: carry the last known value forward until a new measurement appears. Your weight Monday is probably very close to Tuesday's — close enough.

〰️
Step 4: 7-Day Smoothing

I apply a rolling 7-day average to every feature. One terrible Tuesday doesn't tank the whole week's data. This reduces noise while preserving real trends.

🩹
Step 5: Final Gap Fill

Any activity/nutrition NaNs left? Fill with 0. Any physical metric NaNs left? Fill with the average from ALL training participants. No blanks allowed.

📈
Step 6: Relative Change Features

I compute new columns: your 7-day average steps divided by your 30-day average steps. This captures trends (are you getting more active?) not just absolute values.

⚖️
Step 7: Standardization

Finally — I rescale every feature so they all have the same scale. 10,000 steps and 70 kg are very different numbers; the AI treats them equally after this. Crucially: I learn the scale from TRAINING data only, then apply it to test data.

✅
Clean Data

I came in as a messy CSV. I leave as a consistent, complete, standardized table — ready for the AI. Every day accounted for. Every number on the same scale. No blanks.

?

The most important rule: no data leakage

Here's a subtle but critical point. When you standardize data, you need to calculate the average and spread of the numbers. But whose average do you use?

⚠️
The temptation: Calculate the average from all participants (train + test). It's more data, so it seems better.

The problem: The model would indirectly "see" the test participants during training — like a student getting the exam questions before the test. This is called data leakage, and it makes your results look falsely good.
CODE 
scaler = StandardScaler()
scaler.fit(train_df[feature_cols])

train_scaled[feature_cols] = scaler.transform(train_df[...])
test_scaled[feature_cols] = scaler.transform(test_df[...])
val_scaled[feature_cols] = scaler.transform(val_df[...])
PLAIN ENGLISH

Create a scaling calculator (StandardScaler).

ONLY study the training data to learn the average and spread. Never peek at test data here.

 

Apply that training-learned scale to normalize the training set.

Apply the SAME scale to test data — using training stats, not test stats.

Same for validation. Everything calibrated to the training set's reality.

💡
Why this matters for you: If you ever ask an AI assistant to add a new preprocessing step, always ask: "Does this step accidentally use information from the test set?" If yes, you have data leakage. This is one of the most common mistakes in machine learning projects.

03

Time Windows

How 300 days of data becomes thousands of training examples — the sliding window technique.

The core challenge: one score, hundreds of days

Here's the puzzle. Each participant has one cognitive score (taken at the end of the study). But they have 300+ daily rows of fitness data. How do you train an AI on this?

🎞️
The film reel metaphor: Think of each participant's fitness history like a film reel — 300 frames. The AI can't watch all 300 frames at once. Instead, it watches a 90-frame clip at a time. The clip slides forward one frame at a time, and each 90-frame clip is one training window.

Example: One participant with 200 days, window = 90 days, stride = 1

Window 1 (days 1–90)
Day 1
Day 200

→ Window slides forward, creating 110 separate training examples from one participant

💡
Why this matters: Instead of having ~100 participants × 1 score each = 100 training examples, you get 100 participants × ~100 windows each = 10,000+ training examples. The AI gets much more to learn from.

FitnessSequenceDataset — the historian

The file temporal_dataset.py defines a Dataset class called FitnessSequenceDataset. It does three things:

1
Match participants
Find participants who appear in BOTH the fitness data AND the cognitive scores file
→
2
Slice windows
For each participant, slide a 90-day window across their full history
→
3
Return (X, y) pairs
Each window X = 90 days of fitness, y = that participant's cognitive score
CODE 
for end in range(window_days, end_max, stride):
    window = seq[end - window_days : end]
    self.samples.append((window, y, pid))
PLAIN ENGLISH

Loop through all possible end-points for a window, moving one day at a time.

Grab the 90 days ending at this point — the "film clip" ending here.

Save this window + the participant's cognitive score as one training example.

What one training example looks like

Each sample the AI sees is an tensor — a 2D grid of numbers:

# X: the input (90 days × 15 features)
Day 1:[ 0.42, -0.81, 1.23, 0.07, -0.55, ... 15 values ] Day 2:[ 0.51, -0.79, 1.19, 0.11, -0.48, ... 15 values ] ... ( 88 more days ) Day 90:[ 0.89, -0.32, 0.94, 0.55, 0.12, ... 15 values ]
# y: the target (3 cognitive scores)
[ 0.73, 0.61, 0.44 ] # free_recall, nat_recall, temporal_clustering

The AI's job: look at the 90×15 grid of fitness numbers and predict the 3 cognitive scores. Every 90-day window from every participant teaches the model something.

💡
Vocabulary moment: X is called the features and y is called the labels (or targets). In every machine learning project, there is an X and a y. Finding good X features that predict y is half the art of data science.

The lag parameter — the scientific question in code form

Here's where the science gets interesting. The lag_days parameter in the dataset lets you say: "I want the fitness window to END some number of days before the cognitive assessment."

lag_days = 0

Fitness window ends the day of the cognitive test. Current fitness predicts current cognition.

lag_days = 14

Fitness window ends 2 weeks before the test. Did fitness 2 weeks ago predict today's score?

lag_days = 90

Fitness window ends 3 months before the test. Long-term fitness patterns as a predictor.

🔬
Why this matters scientifically: If the model predicts best at lag=14 (2 weeks before), that suggests exercise affects the brain with a roughly 2-week delay. This kind of finding can guide real-world recommendations — "exercise now for brain benefits in two weeks."

04

Teaching AI to Read Time

Two architectures that learn patterns in sequences — and how they're trained, evaluated, and compared.

Why ordinary AI can't read a fitness timeline

A simple neural network sees a set of numbers and produces an output. But it's completely blind to order. It wouldn't know if Monday came before Tuesday or after Wednesday.

🎵
The music sheet metaphor: Imagine someone reads a piece of music — but they see all the notes at once, shuffled randomly. They could tell you which notes are there, but they couldn't play the melody. Fitness data is like music: the sequence and rhythm matter, not just which numbers appear.

This project uses two architectures designed specifically for sequences:

🔄

FitnessLSTM

An LSTM reads the sequence step by step, maintaining a "memory" that it updates as it sees each new day. Like reading a story — each sentence builds on what came before.

🔭

FitnessTransformer

A Transformer reads the whole sequence at once and uses attention to decide which days matter most. Like re-reading a story with the ending in mind — now you know which clues were important.

FitnessLSTM — the step-by-step reader

LSTM stands for Long Short-Term Memory. It processes one day at a time, maintaining a hidden state that carries information forward through the sequence.

CODE 
self.input_proj = nn.Linear(N_FEATURES, d_model)
self.lstm = nn.LSTM(
    input_size=d_model,
    hidden_size=hidden_size,
    num_layers=n_layers,
    bidirectional=True,
)
PLAIN ENGLISH

A translation layer: compress the 15 raw features into 32 richer numbers.

Create the LSTM — the sequence-reading engine.

Each day gets compressed to 32 numbers first.

The "memory" size — how much the model can remember at each step.

Stack multiple LSTM layers for deeper pattern detection.

Bidirectional: read the sequence both forward AND backward — since we have the full window at test time.

End of LSTM setup.

💡
Why bidirectional? Reading a fitness history forward makes sense — earlier fitness affects later fitness. But reading backward too helps the model spot where trends end, not just begin. Since we have the whole window at prediction time (not a live stream), this is safe and useful.

FitnessTransformer — the attention reader

The Transformer architecture doesn't read the sequence step-by-step. Instead, it converts each day into a "token" and then uses attention to decide which days are most relevant to predicting cognition.

The Trainer — teaching the model through repetition

The Trainer class runs the learning loop. Training an AI model is iterative — you show it examples, it makes predictions, you measure how wrong it was, and it adjusts itself slightly. Repeat thousands of times.

📐

MSE Loss

MSE measures how wrong the predictions are. Squaring the errors penalizes big mistakes more than small ones. The model tries to shrink this number every iteration.

🧭

AdamW Optimizer

AdamW decides how to adjust the model's millions of internal numbers after each batch of examples. It's the engine of learning.

🛑

Early Stopping

If the model stops improving on the validation set for 10 epochs in a row, training stops. This prevents overfitting to the training data.

CODE 
if v_loss < best_val_loss:
    best_val_loss = v_loss
    best_state = {...model.state_dict()...}
    patience_counter = 0
else:
    patience_counter += 1
    if patience_counter >= self.patience:
        break

Remember this as the new personal best.

Save a snapshot of the model's weights — we can come back to this.

Reset the patience counter — we're improving, keep going.

05

The Lag Experiment

Does yesterday's run predict today's memory? The scientific heart of this project.

The key scientific question, in code form

The function run_lag_analysis() is the project's scientific core. It runs a full experiment — training and evaluating a model for each possible time gap — to find the answer to one question:

⏳
At what lag does fitness best predict cognition?
If the peak is at lag=0: fitness and cognition are concurrent. Correlation, but not necessarily cause.
If the peak is at lag=30: fitness from 30 days ago best predicts today's cognition. That's a delay — consistent with a causal mechanism.
CODE 
if lag_values is None:
    lag_values = [0, 7, 14, 21, 30, 45, 60, 90]

for lag in lag_values:
    train_loader, ... = make_dataloaders(
        ..., lag_days=lag, ...)
    model = FitnessLSTM(n_tasks=n_tasks)
    trainer.fit(train_loader, val_loader)
    results[lag] = trainer.evaluate(test_loader)
PLAIN ENGLISH

If no lag values specified, use this default list:

Test at 0, 1 week, 2 weeks, 3 weeks, 1 month, 6 weeks, 2 months, 3 months.

 

For each lag value, create fresh data loaders with that time gap applied.

(Windows will end lag days BEFORE the cognitive assessment.)

Create a new model from scratch — no shared weights between lag experiments.

Train the model on training data, watch validation for early stopping.

Record test performance for this lag. Compare across all lags to find the peak.

R² — the score that tells you how well it works

After training, each model is evaluated with a metric called R² (R-squared). Think of it like a golf handicap for model performance:

1.0
Perfect — the model predicts every participant's score exactly right. Never happens in real research.
0.3
Meaningful signal — the model explains 30% of the variation in cognitive scores. For human behavior data, this is genuinely impressive.
0.0
No better than average — the model just predicts the mean score for everyone. Useless.
<0
Actively bad — the model is worse than guessing. Something is wrong.
